# Git Project Management Rules

These rules apply to ALL files. Enforce the team's git conventions automatically.

## Identity (HIGHEST PRIORITY)

- Never include "Generated by AI", "Sent via Claude", or any AI tool signatures in any output
- No AI-related metadata or disclaimers in Issues, PRs, commits, code comments, or docs
- Write as a human developer: concise, professional, no "As an AI..." phrasing

## When creating a branch

- Format: `<type>/<issue-number>-<short-description>`
- Type must be one of: feat, fix, docs, style, refactor, test, chore, ci, perf, revert, experiment
- Description: lowercase, words separated by `-`, 2–4 words max
- Always branch from latest `main`: run `git switch main && git pull` first
- Examples: `feat/12-spd-manifold`, `fix/34-nan-log-map`, `experiment/56-curvature-ablation`

## When writing commit messages

- Format: `<type>(<scope>): <imperative subject>`
- Type: same list as branch types
- Scope must be one of: manifolds, flow, models, layers, data, losses, utils, cli, configs
- Subject: English, imperative mood ("add" not "added"), lowercase start, no period, ≤72 chars
- Footer: use `Refs #N` on intermediate commits; use `Closes #N` only on the final commit before PR
- Body (optional): explain WHY, not WHAT

## When to commit

- One commit = one atomic change that can be described in a single sentence
- Each commit must pass lint (`make lint`) and not introduce syntax/import errors
- **DO commit when:** a single logical unit is complete — a new function, a config addition, a test, a focused bugfix
- **Do NOT commit when:** code is half-finished, multiple unrelated changes are mixed together, or debug residue (`print`, `pdb`, `breakpoint()`) remains
- **Do NOT bundle** unrelated changes: "fix NaN bug + rename variables + update README" = 3 separate commits
- Prefer slightly finer granularity on feature branches — squash merge compresses them into one clean commit on `main`
- If a commit is hard to describe in one `<type>(<scope>): <subject>` line, it's too large — split it

## When creating Issues

- Title format: `[Type] description` where Type is capitalized
- Examples: `[Feat] SPD manifold with affine-invariant metric`, `[Bug] NaN in hyperbolic log map`, `[Experiment] Curvature ablation on FB15k-237`
- **Required labels:** `type:` × 1 + `priority:` × 1
- **Recommended labels:** `scope:` × 1~2 when specific modules are involved
- **Experiment issues:** also add `ml:` × 1 (e.g. `ml:ablation`, `ml:baseline`, `ml:dataset`)
- `status:` labels are dynamic — add/remove as state changes (e.g. `status:blocked`, `status:needs-repro`)

## When creating Pull Requests

- Title format: `<type>(<scope>)[#<issue-number>]: <subject>`
- If no corresponding Issue, omit `[#N]`: `<type>(<scope>): <subject>`
- Examples: `feat(manifolds)[#12]: add SPD manifold`, `chore(ci): add ruff to CI pipeline`
- Body MUST include `Closes #N` to auto-close the linked Issue
- **Required labels:** `type:` × 1 (match the Issue/branch type)
- **Recommended labels:** `scope:` × 1~2 (match the Issue)
- **Do NOT add** `priority:` or `ml:` labels to PRs — these belong at the Issue level
- Add reviewers and milestone
- For ML experiments: include metrics table and wandb link in PR body
- Use Draft PR for work-in-progress

## When merging

- Always use squash merge: `gh pr merge --squash --delete-branch`
- Never merge without at least 1 approval and passing CI
- After merge, clean up local branch: `git switch main && git pull && git branch -d <branch>`

## Code quality gates before push

- `make lint && make format`
- `make typecheck`
- `make test`
- No hardcoded hyperparameters, paths, or device names
- No debug prints, breakpoints, or `import pdb`
- No large files (checkpoints, datasets, model weights)
