# CLAUDE.md

## Project Overview

RiemannFM (Geometry- and Text-Aware Flow Matching on Product Manifolds) is a deep graph generative foundation model that performs joint continuous-discrete flow matching on product Riemannian manifolds (H Г— S Г— E) for knowledge graph generation and downstream tasks (KGC, T2G, GAD).

## Tech Stack

- Python 3.12+, PyTorch 2.x, PyTorch Geometric
- Riemannian optimization: geoopt
- ODE solving: torchdiffeq
- Config: Hydra / OmegaConf
- Text encoding: transformers (Qwen3-Embedding / Sentence-BERT / XLM-RoBERTa)
- Hyperparameter search: Optuna (via hydra-optuna-sweeper)
- Experiment tracking: Weights & Biases (wandb)
- Testing: pytest
- Linting: ruff
- Type checking: mypy
- Package manager: uv (PEP 735 dependency-groups)

## Identity Rules (HIGHEST PRIORITY)

- Never include "Generated by AI", "Sent via Claude", or any AI tool signatures
- No AI-related metadata or disclaimers in Issues, PRs, commits, code, or docs
- No comments like "refactored by assistant" or "AI-generated"
- Write as a human developer: concise, professional, no "As an AI..." phrasing
- Commit messages follow Conventional Commits, never mention AI in subject or body
- PR/Issue descriptions: technical content only, no AI tool links or footers

## Git Conventions

### Branch Naming

```
<type>/<issue-number>-<short-description>
```

Examples: `feat/12-spd-manifold`, `fix/34-nan-log-map`, `experiment/56-curvature-ablation`

### Commit Message Format (Conventional Commits)

```
<type>(<scope>): <subject>

[body]

[Closes #<issue-number> | Refs #<issue-number>]
```

- English, imperative mood, lowercase first letter, no period
- Subject в‰¤ 72 characters
- Body explains WHY, not WHAT

### When to Commit

- One commit = one atomic, describable change ("add SPD manifold", "fix NaN in log map")
- Each commit should pass lint and not introduce syntax/import errors
- Commit when: a single logical unit is complete (new function, config, test, bugfix)
- Do NOT commit: half-finished code, mixed unrelated changes, debug residue
- Granularity: prefer slightly finer commits on feature branches вЂ” squash merge cleans up history
- Never bundle "fix bug + rename variables + update docs" in one commit вЂ” split them

### Issue Title Format

```
[Type] description
```

Type is capitalized. Examples: `[Feat] SPD manifold with affine-invariant metric`, `[Bug] NaN in hyperbolic log map at high curvature`

### PR Title Format

```
<type>(<scope>)[#<issue-number>]: <subject>
```

If no corresponding Issue exists, omit `[#N]`:

```
<type>(<scope>): <subject>
```

Examples: `feat(manifolds)[#12]: add SPD manifold`, `chore(ci): add ruff to CI pipeline`

### Allowed Types

feat, fix, docs, style, refactor, test, chore, ci, perf, revert, experiment

### Allowed Scopes

manifolds, flow, models, tasks, data, losses, optim, utils, cli, configs

### Merge Strategy

Squash merge to main. One feature = one clean commit.

### PR Rules

- At least 1 Approve + CI pass required
- Use Draft PR for WIP / early feedback
- PR body must reference issue: `Closes #N`
- Include training metrics / experiment results when applicable

### Label Policy

**Issue (required):**

- `type:` Г— 1 вЂ” always (e.g. `type:feat`, `type:bug`, `type:experiment`)
- `priority:` Г— 1 вЂ” always (e.g. `priority:high`, `priority:medium`)

**Issue (recommended, add when applicable):**

- `scope:` Г— 1~2 вЂ” when specific modules are involved
- `ml:` Г— 1 вЂ” for experiment-type issues (e.g. `ml:ablation`, `ml:baseline`)
- `status:` вЂ” dynamic, add/remove as state changes (e.g. `status:blocked`)

**PR (required):**

- `type:` Г— 1 вЂ” match the corresponding Issue/branch type

**PR (recommended):**

- `scope:` Г— 1~2 вЂ” match the Issue for filtering convenience

**PR does NOT need:** `priority:` or `ml:` (these belong to the Issue level)

**Release / Tag:** no labels needed, use SemVer tags and Milestones instead

## Code Conventions

- Class naming: project classes use `RiemannFM` prefix (`RiemannFMProductManifold`, `RiemannFMJointFlow`, `RiemannFMRieFormer`, `RiemannFM`)
- Math single-letter variable names are allowed (ruff N806/N812 suppressed): `N` for batch size, `B` for batch, `D` for dimension, `K` for components, `F = nn.functional`, `M` for manifold dimension
- Type hints on all function signatures; use `X | Y` union syntax, `list[T]`, `dict[K, V]` (PEP 604/585)
- Docstrings: Google style with Args/Returns/Raises; document tensor shapes (e.g. `shape (..., dim)`)
- File names: snake_case
- Constants: UPPER_SNAKE_CASE
- Config keys: snake_case
- No wildcard imports (`from module import *`)
- No hardcoded paths, hyperparameters, or magic numbers вЂ” use Hydra configs
- Device-agnostic code: never hardcode `cuda:0`, use accelerator abstraction
- Reproducibility: always set and log random seeds

### Data Pipeline

```bash
uv run python -m riemannfm.cli.download data=wikidata_5m
uv run python -m riemannfm.cli.download download.all=true
uv run python -m riemannfm.cli.preprocess data=wikidata_5m preprocess.build_mini=true
uv run python -m riemannfm.cli.preprocess data=wikidata_5m_mini embedding=sbert
```

### Pretraining (Hydra overrides as CLI args)

```bash
uv run python -m riemannfm.cli.pretrain training.lr=1e-4 data=fb15k237
uv run python -m riemannfm.cli.pretrain model=rieformer_small data=wikidata_5m_mini training.max_steps=1000
```

### Logger Selection (Hydra config group)

```bash
uv run python -m riemannfm.cli.pretrain logger=default      # wandb + csv (default)
uv run python -m riemannfm.cli.pretrain logger=wandb_only    # wandb only
uv run python -m riemannfm.cli.pretrain logger=csv_only      # csv only (offline)
uv run python -m riemannfm.cli.pretrain logger=none           # no logger
```

## Before Committing

- Run `make precommit` (or individually: `make lint`, `make typecheck`, `make test`)
- Remove debug prints, breakpoints, and notebook artifacts
- Ensure no large files (checkpoints, datasets) are staged
- Commit footer: use `Refs #N` on intermediate commits; use `Closes #N` only on the final commit before PR

## Experiment Discipline

- Every training run must be logged to wandb with a descriptive run name
- Config must be committed BEFORE running the experiment
- Record: model arch, dataset version, hyperparams, hardware, final metrics
- Tag reproducible results: `git tag exp/<experiment-name>`
- Report results with 3 seeds (42, 123, 456) as mean В± std
